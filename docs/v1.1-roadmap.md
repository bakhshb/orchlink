# Orchlink v1.1 Roadmap

This roadmap records what v1 of the coordination-kernel hardening (G005)
deliberately defers, and why. It is the companion to
`docs/coordination-kernel-implementation-plan.md` (see M5 and M6).

## v1 boundary (delivered)

- Task/talk/session **live state is broker/API-driven**. The CLI obtains it
  through `BrokerClient` (HTTP), not by reading `.orch/` files directly.
  `tests/test_cli_no_live_fs_coupling.py` guards this: task/talk CLI modules
  must not reference `.orch/goals/*` live-state paths, may reference
  `.orch/run/*` only for the broker store-path default (plumbing), and must
  not import the Goal Mode live-state layer (`orchlink.goal.{store,runner,
  checks,models}`).
- Goal Mode **stays file-coupled** in v1. `goal/store.py` continues to read
  and write `.orch/goals/<id>/{goal.yaml,history.jsonl,acceptance.md,...}`
  directly. This is intentional and in scope for the guard (the guard targets
  task/talk CLI only).

## Deferred to v1.1: Goal Mode projection (M6)

Make `.orch/goals/*` live state a **projection** of the broker audit journal
rather than a parallel store.

- `goal.yaml`, evidence, blockers, and deferred-state are derived from the
  journal.
- The goal CLI reads live goal state through broker APIs instead of files.
- Authored artifacts stay file-based by design: `source.md`,
  `acceptance.md`, `plan.md`, `coverage.md`. These are human-authored inputs,
  not live state, and remain on disk after projection.

### Why deferred

- v1 is single-operator; goal file-coupling has no concurrency cost yet.
- Projection (M6) is the riskiest milestone. Shipping it separately avoids
  coupling the v1 release to it.
- v1 audit journaling (M1) captures `goal.*` transitions from day one, so M6
  needs **no backfill migration**: the projection reads the journal and
  derives state. This is the one piece of v1.1 plumbing that v1 deliberately
  front-loads to keep the future migration cheap.

## Permanent contracts (not deferred)

- **Best-effort cancellation.** Orchlink revokes leases, sends steering, and
  aborts if Pi exposes an abort hook. It does **not** guarantee that
  already-running tool calls stop immediately. This is the permanent public
  contract for v1 and v1.1; no stronger guarantee is planned.
- **JSONL storage** for the local-first single-operator model. No SQLite swap
  is planned for "production maturity."

## Open v1.1 questions (not decided here)

- Whether to route goal *writes* through the broker before full projection
  (a partial step that would journal writes centrally).
- Whether `doctor` should report goal/journal projection health.
